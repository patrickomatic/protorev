# Agent Notes

## Project

`protorev` is a Rust CLI and library for reverse engineering raw protobuf wire
streams. Treat the installed command name as `protorev`; README examples should
not be written as `cargo run -p protorev -- ...`.

## Common Commands

- `cargo test` runs the full test suite.
- `cargo clippy --all-targets --all-features` checks the configured lints.
- `cargo fmt` formats the workspace.
- `dist plan` validates cargo-dist release metadata when `cargo-dist` is
  installed.
- `dist build --target x86_64-apple-darwin` is a useful local smoke test on
  macOS for cargo-dist packaging.

## Coding Guidelines

- Keep the crate dependency-light; do not add dependencies for small parsing or
  formatting tasks without a clear payoff.
- The manifest denies `unwrap`, `expect`, and `panic` through clippy. Propagate
  errors through the crate's `Error` type instead.
- Preserve offset and wire-type evidence in user-facing output. The tool should
  report observations, not invent semantic protobuf meaning.
- Prefer deterministic ordering in text and JSON output so corpus comparisons
  and tests stay stable.

## Tests And Fixtures

- Public fixture inputs live under `fixtures/public`.
- If behavior changes, update or add focused tests under `tests/` rather than
  relying on manual CLI runs.
- Keep fixture data small and inspectable. Hex fixtures are preferred for
  protobuf samples that should be easy to review in git.

## Releases

- Releases are managed with cargo-dist using `dist-workspace.toml`.
- The GitHub release workflow is generated by cargo-dist; regenerate it with
  `dist generate` after changing cargo-dist config.
- Do not move an already-published crates.io version or GitHub release tag.
  Bump `Cargo.toml` and create a new tag instead.
